      <h2><a name="session_366"></a>Where No One Has Tested Before: The Case For Textless Environments</h2>
      <h3>How to test-drive your code, in a language as foreign as GLSL</h3>
      <h3>
          <a href="#presenter_396">Emmanuel Gaillot</a>
      </h3>
    </td>
  </tr>
  <tr><td colspan="2" ><p>We&apos;ll show how we can use TDD to build a non-trivial shader in GLSL (OpenGL Shading Language). This tutorial is intended to push the limits of what is considered doable with TDD. This is a performative programming session: the audience will watch a performer programming for its (the audience&apos;s) benefit some non-trivial piece of code from A to Z. Some people like to watch by doing themselves. If you belong to this strange crowd, bring a laptop. But really this isn&apos;t mandatory or even encouraged.</p></td></tr>
  <tr>
    <td class="propertyname">Goal of the session: </td><td>1) to learn how to test-drive one&apos;s developments in contexts where there is no tool, framework, or even text output;2) to get a taste about all kinds of cool graphics you can render with a GPU;3) to inspire others running live coding sessions themselves, and thus contributing honing the skills of the programmers&apos; community;4) to meet programmers who enjoy exotic languages;5) to have fun watching someone showing off.  </td>
  </tr>
    <tr>
      <td class="propertyname">Intended audience:</td><td> Jan, Leo, Bram and Hank.</td>
    </tr>
    <tr>
      <td class="propertyname">Expected experience:</td><td> This session is intended for developers of all confessions, aspiring developers of all ages, and professionals of all trades who wish they were still developers. Attendees should have at least some theoretical understanding of what TDD is, and a practical experience at programming software. Knowledge ofnon-aristotelian drama theory isn&apos;t required. </td>
    </tr>
  <tr>
    <td class="propertyname">Session Type:</td><td> 75 min hands on coding/design/architecture session </td>
  </tr>
  <tr><td colspan="2" ><p>Test-Driven Development is desperately simple. Simple, because it takes about five minutes to understand the principles. Desperately, because real-life contexts seem always harder than case studies used for training -- at least by forty-two orders of magnitude.</p><p>Just for laughs, we&apos;ll aim this time at doing the opposite (to practice TDD in a more difficult context than the typical workplace) and we&apos;ll show how we can use TDD to build a non-trivial shader in GLSL (OpenGL Shading Language). This tutorial is intended to push the limits of what is considered doable with TDD. By doing so, we hope to make the audience realize that if it&apos;s possible to test-drive code in GLSL, it should be possible to do so in virtually any TDD-resistant environment.</p><p>This is a performative programming session: the audience will watch a performer programming for its (the audience&apos;s) benefit some non-trivial piece of code from A to Z.  In the spirit of the Brechtian theatre, the audience is encouraged to laugh at the misery of the programmer on the stage, to be critical of their (the programmer&apos;s) successes, to cheer and to boo, to smoke and to eat, and to think about what they&apos;d do (better) if they were in the same position.</p></td></tr>

      <h2><a name="session_381"></a>Ice breakers, powerful tools for teambuilding</h2>
      <h3>Teamwork is a great thing, but it is not always so easy to be part of a team...</h3>
      <h3>
          <a href="#presenter_410">Giovanni Puliti</a>
      </h3>
    </td>
  </tr>
  <tr><td colspan="2" ><p>Ice breakers are a powerful tool to <b>facilitate</b> the <b>team working</b>. In this session we will see some exercises and games to help the <b>creation</b> of the groups, <b>collaboration</b>, <b>communication</b>, or even just to <b>break the ice</b> and create a close-knit group. 
<br />Once a student in a class said: <i>&quot;thanks to the exercises that you have made us , I found myself working with others like I know them for a long time , even if we met this morning for the first time &quot;</i> The goal of this session is exactly this.</p></td></tr>
  <tr>
    <td class="propertyname">Goal of the session: </td><td>have fun with a lot of games, exercises, plays to learn something about how we interact, how we stay togheter, how we think...  </td>
  </tr>
    <tr>
      <td class="propertyname">Intended audience:</td><td> Jan, Marieke, Leo, Bram, Philippe, Joke, Hank, Ellen, </td>
    </tr>
    <tr>
      <td class="propertyname">Expected experience:</td><td> no experience required </td>
    </tr>
  <tr>
    <td class="propertyname">Session Type:</td><td> 75 min experiential learning session </td>
  </tr>
  <tr><td colspan="2" ><p>Agile methodologies give great importance to the concept of the <b>team</b>, to being part of a group where people work together to achieve goals.
<br /><b>Teamwork</b> is a great thing, but it is not always so easy to be part of a team.</p><p>Tuckman through its <b>five stages model</b> (forming, storming, norming, performing positions) explained that, before reaching the stage performance, a team must pass a series of steps related to the growth of the team itself.
<br />As <b>agile coach</b> often find myself having to <b>facilitate</b> and <b>simplify</b> the process of growth to get in the easiest and fastest in the performing stage.</p><p>Due to my experience in the <b>theater of improvisation</b>, in <b>public speaking</b>, <b>agile coaching</b>, creating <b>communities</b> within organizations, I have collected a number of useful tools such as <b>ice breakers</b> and then simplify the phase of <b>team building</b>.
<br />I use them in my classes, but also in the coaching of development teams and in general all the times when you need to find a way to help people know each other. My set of ice breakers embraces issues of serious gaming, time management and time awarness, comunication and public speaking skills, agile management team and more.
<br />At the end of these exercises in the group circulated a <b>positive energy</b>, <b>joy</b> and the <b>desire</b> to work together. 
<br />I would like to share them with you.</p></td></tr>

  <tr><td colspan="2" ><p>The TDD cycle states: write a test, see it fail, make it work, refactor. We&apos;re guided by principles like baby steps and &quot;Do the simplest thing that could possibly work&quot;. But how do you find the simplest next step to make the test pass? How can you find the next baby step and prevent the apparently inevitable big leap you sometimes have to make? Is there more to it than gut feel and intuition?</p><p>There is a set of transformations you can apply to get from Red to Green. These have names, but more importantly, there seems to be a logical order. If you obey this order, you are less likely to get stuck and more likely to arrive at simpler, more efficient code.</p><p>In this workshop, we&apos;ll introduce these transformations and let you experiment with them hands on.</p></td></tr>

  <tr><td colspan="2" ><p>You might know the feeling: just as you&apos;re doing great with your tests, you run into a situation where you&apos;re unable to get the next test working without structural change to the production code. Or you have finished your test, you inspect the test coverage, and find out that again you wrote production code without a test that requires it. Can you prevent these situations?</p><p>&quot;Getting to green&quot; is about the Red-Green-Refactor cycle from test driven development. To get from Red to Green, we want to take baby steps. At each step a small, simple transformation should be sufficient to get from red to green. </p><p>This session explores the ideas and transformations that were introduced by Robert C. Martin in his <a href="https://blog.8thlight.com/uncle-bob/2013/05/27/TheTransformationPriorityPremise.html"> Transformation Priority Premise</a> article.</p><p>His premise is that 1) there is a set of well-defined transformations to make failing tests pass and 2) there is a logical order in which to apply these transformations: by taking this order into account, you should be able to avoid big steps and impasses.</p><p>Examples of the transformations are: going from no code to nil/null, going from nil to a constant, going from unconditional code to an if. They are in a way similar to refactorings (but refactorings keep tests green and the transformations make a test green).</p><p>In this workshop, we are going to investigate the premise of the transformations and their logical order. We introduce the transformations using a coding kata. After the introduction, you will get to work on another kata. Bring your own device, language and development environment. You&apos;ll write all the code yourself or you&apos;ll pair up with another participant.</p><p>Learning about these transformation will help you improve your test driven development and design skills. It will also (possibly) help you in finding baby steps to proceed. As a community, we hope to learn more about structured ways of getting from red to green.
